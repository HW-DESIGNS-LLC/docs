nRF5340 BLE Dev Board

A dual‑core Bluetooth LE development board with a built‑in debug probe — no J‑Link needed — plus USB‑C, LiPo battery charging, a Qwiic connector, and a dedicated current‑measurement header for true µA power profiling.

Getting started

From the box to your first flash

The board arrives ready to use — the debug probe is built in, so there is nothing to buy and no drivers to install. Steps 1–2 show the board working; steps 3–4 set up the nRF Connect SDK and flash your own firmware.

  1. 1

    Plug it in

    Connect the board to any USB‑C charger or computer port. It powers up and starts the built-in demo firmware, advertising over Bluetooth LE.

    The DEBUG / APP slide switch routes the USB‑C port. Either position powers the board — the switch only chooses which chip the data lines reach (step 3 explains).

  2. 2

    See it from your phone

    Install the free nRF Connect for Mobile app (Android/iOS) and scan. The board appears as HWD‑5340 — tap it to watch live signal strength, and connect to browse its services.

  3. 3

    Install the nRF Connect SDK

    This is a standard nRF5340 target, so Nordic’s official toolchain is the path: install VS Code and the nRF Connect for VS Code extension pack, then use its welcome page to install a toolchain and SDK. Build for these targets:

    SDKnRF Connect SDK v2.9 or later
    Board targetnrf5340dk/nrf5340/cpuapp
    Build systemSysbuild (builds the app + network cores together)
    LFXO load capsInternal 6 pF — add CONFIG_SOC_LFXO_CAP_INT_6PF=y to prj.conf

    The board is pin-compatible with the official nRF5340‑DK for uart0, i2c1, and the QSPI flash — stock DK examples build and run as-is. The one required change is the LFXO line above (this board uses the crystal’s internal load capacitors).

  4. 4

    Flash your first program

    Slide the switch to DEBUG — the onboard probe enumerates as a CMSIS‑DAP debugger with a serial port. Install probe‑rs, build any sample (Blinky is a good first one), and flash from the terminal:

    west build -b nrf5340dk/nrf5340/cpuapp --sysbuild
probe-rs download --chip nRF5340_xxAA --binary-format hex build/merged.hex
probe-rs reset --chip nRF5340_xxAA

    Flash from the terminal, not the VS Code buttons. The extension’s Flash/Debug buttons expect a J‑Link and won’t see the onboard CMSIS‑DAP probe. Terminal flashing (probe‑rs, west flash --runner pyocd, or OpenOCD) works with everything, and debugging works through the Cortex‑Debug extension.

    First connect says the device is locked or protected? That’s the nRF5340’s access-port protection on a fresh chip — run probe-rs erase --chip nRF5340_xxAA --allow-erase-all once to recover it, then flash again.

  5. 5

    Keep going

    Take the board tour below, grab the pinout diagram and user guide, then build something from the example projects — including the power-profiling walkthrough that shows off what this board does best.

Board tour

Know your board

Both sides, and what everything on them does.

Front

  • MDBT53‑1M module — the exposed end is the Bluetooth antenna; keep it clear of metal.
  • DEBUG / APP slide switch — routes USB‑C to the onboard debug probe or to the nRF5340’s own USB.
  • RESET button for the nRF5340, plus BOOT and RST for the debug-probe side (used only for probe firmware updates).
  • USB‑C for power, charging, programming, and serial.
  • CHG and power indicator LEDs, plus a user LED under firmware control.
  • Qwiic / STEMMA QT connector (JST‑SH 4‑pin) for solder-free I²C sensors.

Back

  • Every header pin labeled on the silkscreen — power (3V3, GND, 5V), port/pin numbers, and TX/RX.
  • J2 current-measurement header — the jumper feeds the nRF’s isolated rail. Leave it fitted for normal use; swap it for an ammeter to profile power.
  • LiPo battery connector (JST‑XH) with polarity marked + / −.
  • Tag‑Connect TC2030 pads — direct SWD access for an external J‑Link or probe, no header needed.
  • NFC antenna pads for an optional NFC‑A tag antenna.
  • Board revision code printed along the edge — include it when you contact support.

Measuring microamps?

The nRF5340’s supply rail is isolated and fed only through the J2 header. Jumper on = normal use. Jumper off, ammeter in its place = you are measuring exactly what the nRF and its flash draw — down to sleep currents — while USB, the debug probe, and the Qwiic port stay powered on their own rail.

Best Bluetooth range

Let the antenna end hang over the edge of your breadboard or enclosure. Metal, batteries, and hands near the antenna reduce range.

Before you plug in a battery

Use a single-cell 3.7 V LiPo with a JST‑XH plug and match the + / − marks on the board. Battery lead polarity is not standardized between vendors — check every new battery.

My computer doesn’t see the board

Check the DEBUG / APP slide switch first — it routes the USB‑C data lines. DEBUG shows the onboard probe (a CMSIS‑DAP debugger plus a serial port); APP shows the nRF5340’s own USB, which only appears if the running firmware enables it. Also use a data-capable USB‑C cable — many bundled cables are charge-only. No drivers are needed on Windows 10/11, macOS, or Linux.

Which battery does it take?

A single-cell 3.7 V LiPo with a JST‑XH plug. It charges automatically from USB‑C at ≈200 mA, and on-board protection guards against over-charge, over-discharge, over-current, and reverse polarity. Match the + / − marks next to the connector — battery lead polarity varies between vendors.

Do I need a J‑Link?

No — the debug probe is built in. Flash and debug from the terminal with probe‑rs, pyOCD, or OpenOCD, and use the Cortex‑Debug extension in VS Code for breakpoints. (The nRF Connect extension’s Flash/Debug buttons expect a J‑Link, so they won’t see the onboard probe.) If you prefer your own J‑Link, the Tag‑Connect TC2030 pads on the back give it direct SWD access.
